# 4.6 — Progress Reporting (Celery + Modal)

**Date:** 2026-06-24
**Status:** Complete — 40 tests passing in the `ProPredict` conda env

## Goal

Replace the hardcoded 0/50/100 job progress with real per-stage progress, for
**both** execution backends (Celery and Modal).

## Design — one callback, two transports

`_run_prediction_core` is shared by Celery and Modal and has no `self`. Rather than
branch inside it, it now takes an **optional** `progress_cb(percent, stage)` and
emits at coarse stage boundaries. Each wrapper supplies a callback wired to its own
transport:

| | Celery | Modal |
|---|---|---|
| Worker has... | `self` (`bind=True`) | no Celery context |
| Transport | `self.update_state(state="PROGRESS", meta=...)` → Redis result backend | `modal.Dict` named `propredict-progress`, keyed by `run_id` |
| API reads via | `AsyncResult(run_id).info` | `modal.Dict.from_name(...).get(run_id)` |

**Why a `modal.Dict` and not Postgres for Modal:** the Modal worker runs in Modal's
cloud and can't be assumed to reach the (local/compose) Postgres, which is likely why
the existing design has the API store `modal_call_id` and poll the `FunctionCall`
rather than the worker writing to the DB. `modal.Dict` is Modal-managed shared state
reachable from both the worker and the API, so it sidesteps DB reachability entirely.

## Stage boundaries

Emitted from `_run_prediction_core` (constants in `orchestrator/progress.py`):

- `10` `folding` — start of initial multi-seed predictions
- `40` `post_processing` — after best prediction selected + inter-model compare
- `60` `simulation` — only when MD actually runs
- `90` `finalizing` — before assembling the result dict
- `100` is implicit: Celery sets `SUCCESS` / Modal's `FunctionCall` returns.

`_emit` wraps the callback in try/except so a status-backend hiccup can never break
the actual prediction (covered by a test).

## Status endpoint

`get_job_status` now uses the pure helper `celery_state_to_status(state, info)`
(in `orchestrator/progress.py`) for the Celery branch, and `_read_modal_progress`
for the Modal branch (refines the in-flight 50% with the Dict's percent/stage).
`JobStatus` gained an optional `stage` field so the API surfaces the human-readable
stage alongside the percent.

FAILURE safety: the helper only reads `info` when state is `PROGRESS`; on FAILURE
Celery sets `info` to the exception, which is never dereferenced (tested).

## Decisions / deviations

- **`modal.Dict` for Modal progress** (chosen over keeping Modal coarse) — user opted
  for symmetric per-stage progress on both backends.
- **`celery_state_to_status` lives in `orchestrator/progress.py`**, not `api/main.py`,
  so it's unit-testable without triggering `init_db()` (importing `api.main` requires
  Postgres). Import-light module, shared by worker + API.
- **Optional `progress_cb`** keeps the dict-in shared-entry invariant intact: Modal's
  old `_run_prediction_core(request_data)` call still works; only the wrappers changed.
